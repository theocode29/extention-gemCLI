# Plan for Implementing the Gemini CLI Datapack Dev Agent Extension

## Goal
Transform the current Gemini Datapack Architect extension to match the master plan for the "Datapack Dev Agent" extension, preserving only the agentic multi-agent system (planner, developer, porter, critic) and the plan-before-code workflow.

## Key Principles to Preserve
- Agentic architecture with multiple specialized agents
- Stateless AI requiring external state management
- Plan-before-code execution flow
- Multi-step agent collaboration

## Overview of Changes
**Keep:**
- Agentic multi-agent system (Planner, Developer, Porter, Critic)
- State persistence mechanism (.gemini-project.json, .gemini-mission.json/.md)
- Research gate and contradictory duel (Planner vs Critic)
- Technical contract gate
- Human-in-the-loop validation gates (3 gates)
- Capability-first transverse engine
- Manifest-based capability catalog

**Replace/Redesign:**
- MCP Server implementation to match strict tool signatures from master plan
- Shadow Git system for non-destructive edits
- Surgical file editing tools (read_file, create_file, edit_file_replace, edit_json_value, rollback_last_action)
- Smart Linter with Spyglass and 3-layer filtering
- Library Registry with .libmanifest.json and capability routing
- Playbook (system prompt) with strict rules from master plan
- Project scaffolding tool
- Task state management (.gemini_tasks.json)
- Auto-Manifest Generator maintenance tool

## Implementation Roadmap (Adapted from Master Plan)

### Phase 1: Foundation & Surgical Editing (Weeks 1-2)
1. Develop base Node.js MCP server
2. Implement `scaffold_project` with Shadow Git initialization
3. Implement surgical editing toolkit:
   - `read_file` (with line numbers)
   - `create_file`
   - `edit_file_replace` (strict search & replace)
   - `edit_json_value`
   - `rollback_last_action` (git reset --hard HEAD~1)
4. Pre-hook: Automatic `git add . && git commit` before every modification tool
5. Unit tests for file operations and git integration

### Phase 2: Smart Linter (Week 3)
1. Integrate `@spyglassmc/cli` as subprocess
2. Parse Spyglass JSON output
3. Implement 3-layer filter:
   - Path exclusion (library directories)
   - Semantic amnesty (macro calls like `$function`)
   - Severity distinction (ERROR vs WARNING)
4. Return filtered results to AI in specified format

### Phase 3: Library Router (Week 4)
1. Define `.libmanifest.json` schema
2. Implement capability scoring algorithm:
   - `list_capabilities()`: aggregated, deduplicated list
   - `request_capability(capability, context_tags)`: tag-based matching
   - `link_library(library_name)`: physical installation + automatic load/tag injection
3. Create manifests for Bookshelf and BONE:MSD
4. Implement spyglass ignore paths from manifests

### Phase 4: Orchestration & HITL (Week 5)
1. Implement task state management:
   - `read_task_state()`: reads `.gemini_tasks.json`
   - `update_task_state(state_json)`: updates todo list
2. Implement `request_human_review(reason, warnings)`: pause MCP, await user input
3. Finalize Playbook (system prompt) with strict rules:
   - Start each iteration with `read_task_state()`
   - Only use `scaffold_project` for project structure
   - Never rewrite entire files; use surgical edits
   - Use capability routing for libraries
   - Immediately fix linter ERRORS
   - Address linter WARNINGS or request human review
   - Never manage load/tag libraries manually; use `link_library`
4. Write final system prompt (.md) for Gemini CLI

## Risks & Attention Points (From Master Plan Section 9)
1. **Surgical Edit Sensitivity**: `edit_file_replace` must return explicit error if snippet not found
2. **Macro Typing**: No reliable type checking; rely on human review for macro correctness
3. **Token Leakage**: Library snippets must be concise (code + max 3 lines explanation)
4. **Error Bail-out**: After 3 failed correction attempts, force `request_human_review`

## Success Criteria
- AI agent follows the strict playbook rules without deviation
- All project modifications go through Shadow Git with automatic commits
- Library capabilities are routed correctly via manifest system
- Linter errors block progression; warnings are addressed or escalated
- Human review gates function correctly at plan, assets, and delivery stages
- Extension can scaffold, develop, lint, and link libraries for Minecraft 1.20.2+ datapacks
